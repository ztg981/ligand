# Ligand private MCP threat model

Last reviewed: 2026-07-14

Status: controls implemented locally; real OAuth and preview evidence still required.

## Security objective

Let one pinned ChatGPT OAuth client act for one pinned Ligand user on only the task records the user explicitly selects. Journal entries, notes, reflections, recovery goals, wellness data, settings, backups, authentication data, and raw synced documents must remain unavailable. The initial deployment is read-only. A later test-only stage may add, complete, or reschedule tasks; it can never delete them.

## Assets and trust boundaries

Protected assets:

- the Ligand account and OAuth grant;
- all raw `user_data` content;
- authoritative task records, versions, and selected goal scope;
- deployment configuration and public client identifiers;
- operational logs and audit metadata.

Trust boundaries:

1. ChatGPT sends an MCP request and a Supabase-issued access token to the public Vercel gateway.
2. The gateway verifies the token and forwards it with only a publishable Supabase key.
3. Supabase RLS and narrow security-definer RPCs re-check client, user, resource, category, and goal scope.
4. Direct Ligand clients reconcile the authoritative task records into the legacy JSON/offline cache.

The gateway has no OpenAI API key, Supabase service-role key, database password, or other bypass credential.

## Threats and controls

| Threat | Primary controls | Residual risk / required evidence |
| --- | --- | --- |
| Stolen, forged, expired, or wrong-audience token | Signature verification through Supabase claims; exact issuer, audience/resource, expiry, not-before, `client_id`, user ID, `ligand_mcp` marker, and `openid` scope checks in the gateway and database | Verify real issued-token rejection in a preview; revoke the grant immediately after any suspected token theft |
| Another OAuth app acts as Ligand | Exact client ID is pinned in Vercel, the consent page, token hook, and `assistant_oauth_clients` | Dynamic registration must be disabled after bootstrap if the existing client continues to work |
| Cross-user or confused-deputy access | Gateway pins one user; OAuth allowlist binds client/resource/user; every row/RPC is user-bound | Test a second Supabase user with a real token |
| OAuth redirect or scope manipulation | Consent accepts only the current user, exact client, HTTPS `chatgpt.com/connector/...` redirect, and exactly `openid` | Review the production callback generated by ChatGPT before approval |
| Raw-document or sensitive-category disclosure | OAuth clients are blocked from `user_data`; named RPCs return strict projections; goal scope is allowlisted; recovery goals are hard-excluded even if selected accidentally | Inspect preview logs and real tool output; never add a general SQL/search/raw-JSON tool |
| Prompt injection stored in task text | Task text is returned as data only; tools are closed-world and cannot browse or execute arbitrary instructions; no task content changes access scope | ChatGPT can still reason badly about malicious text; retain “Always ask” for writes and require an unambiguous user request |
| Ambiguous task mutation | Completion/rescheduling require the exact task ID and current version returned by `get_tasks`; no name-search write exists | The model must re-read or ask the user after a conflict; it must not guess a new version |
| Duplicate/replayed mutation | Every write requires a bounded idempotency key; identical retries replay; a key reused with different arguments is rejected | Retain idempotency metadata; monitor unusual replay/rate patterns |
| Stale device overwrites an assistant change | `task_records` is authoritative; optimistic versions reject stale writes; the app writes only explicitly queued mutations; stale cache omissions and edits are repaired from records | Test two upgraded clients plus an assistant write in a preview before enabling write flags |
| Direct client bypasses version checks | Authenticated users have table `SELECT` only; record mutations go through `task_record_apply` | Confirm deployed grants after migration |
| Accidental or model-driven deletion | No MCP delete tool or assistant delete RPC exists; assistant-created changes use soft-deletable records only through direct Ligand UI workflows | Keep schema/tool-list regression tests in CI |
| Excessive calls or cost/availability abuse | Per-user/client/tool read limits; lower write limits; bounded schemas and output sizes | Vercel/Supabase platform limits and monitoring are still required |
| Secrets or personal content in logs | No privileged gateway secret; audit rows contain IDs/action/outcome/count/request ID only; idempotency stores only a digest, task ID, and version | Inspect Vercel and Supabase logs with harmless test data before production |
| Misconfigured staged rollout | Write tools default off in the server; write consent defaults hidden; database writes default false per user; all three gates are required | Treat any disagreement between flags as a failed deployment and keep server flag false |

## Authorization invariants

- OAuth consent alone never grants data access; an enabled `assistant_access` row and selected task scope are also required.
- `tasks_write` implies an enabled read scope but is an independent opt-in.
- An unassigned task is accessible only through its separate toggle.
- A selected recovery goal remains inaccessible.
- OAuth sessions cannot read task tables or assistant settings directly; only the named RPCs may project data.
- Direct Ligand sessions cannot bypass record versions with table DML.
- A write conflict is a successful safety outcome: the authoritative record is returned and no overwrite occurs.

## Staged go/no-go gates

Read-only preview may begin only when:

- all migrations run on a disposable project/branch;
- the gateway lists only `get_tasks` with both write flags false;
- real tokens for the wrong user/client/resource and expired tokens are rejected;
- the OAuth token cannot access raw tables;
- one harmless selected goal returns expected fields and all hard-denied content remains absent;
- logs and audit rows contain no task/goal text.

Test-only writes may begin only after the read-only trial is stable and:

- all active clients have record reconciliation;
- two-client stale edit, omission, deletion, and foreground-repair cases pass;
- both deployment flags are intentionally enabled together;
- the user reconnects and separately selects limited task changes;
- add replay, stale-version conflict, completion, rescheduling, recovery denial, rate limiting, revocation, and no-delete checks pass.

Production promotion is a human decision. Failing any gate means keep or return the server write flag to false and revoke the OAuth grant if authorization behavior is uncertain.
